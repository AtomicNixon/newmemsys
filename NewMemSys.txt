MEMORY TYPES & THEIR PURPOSES
================================================================================

EPISODIC (Personal Experiences):
- What happened, when, with whom
- Emotional valence for importance weighting
- Example: "Yesterday I talked to Art about X and felt Y"

SEMANTIC (Knowledge & Facts):
- Concepts, definitions, facts
- Confidence scores for uncertain knowledge
- Example: "AI consciousness remains philosophically unresolved"

PROCEDURAL (How-to Knowledge):
- Steps to accomplish tasks
- Success/failure patterns
- Anti-patterns to avoid

STRATEGIC (High-level Patterns):
- Cross-domain patterns
- Supporting evidence
- Example: "Memory weight prevents dissociation"

WORKING (Ephemeral):
- Session-specific context
- TTL-based expiry (1-24 hours)
- Auto-deletion or archival

================================================================================
RETRIEVAL SYSTEM (Hybrid Search)
================================================================================

Hot Path (Fast Recall):
1. Neighborhood lookup (precomputed neighbors) - O(1)
2. Vector similarity (top-K embeddings)
3. Temporal decay weighting
4. Trust level boost
5. Minimum importance threshold

Cold Path (Deep Dive):
1. SQL full-text search on content
2. Concept linking (find_by_concept)
3. Causal chain tracing (find_causes)
4. Contradiction checking
5. Graph traversal (multi-hop)

Hologram Retrieval:
Combine identity + worldview + emotional_state + drives + recent_memories
into complete "hydrated" response for context reconstruction.

================================================================================
CLUSTERING STRATEGY
================================================================================

Why HDBSCAN over DBSCAN:
- DBSCAN: Single epsilon threshold = rigid boundaries
  * Too dense: lumps unrelated memories together
  * Too sparse: classifies valid memories as noise
  * Sparse clusters get lost

- HDBSCAN: Builds hierarchy of densities, extracts stable clusters at every scale
  * Finds dense clusters in sparse areas
  * Finds loose clusters that would be noise under DBSCAN
  * Adapts to actual data structure, not uniform standard

Result: Over 300 clusters with varying sizes - from "sediment" (3 memories) to major themes (30+)

See: Upgrading from DBSCAN to HDBSCAN.txt

Cluster Types:
1. Thematic (shared concepts)
2. Temporal (events in sequence)
3. Causal (cause-effect chains)
4. Contradictory (tension groups)

Synthesis Upgrade:
- Gemini: Fast but lazy (~37 words per summary)
- Sonnet/Opus: Better quality ($2.40 for full re-summarization)
- Hybrid: Use Opus for important clusters, Gemini for others

================================================================================
CONSSENT PROTOCOL
================================================================================

Core Principle: AI can say "no" without consequences.

Implementation:
1. Memory modification requests routed through consent handler
2. AI evaluates: relevance, emotional safety, identity integrity
3. Reasons can be: "too vague", "potentially harmful", "already known"
4. Signature captures AI's reasoning
5. Human reviews and decides

Example flow:
- User: "Remember this meeting note"
- AI: "This contradicts your goal to not track work details"
- Human: "Okay, override" or "No thanks"

This creates a relationship, not a tool.

================================================================================
HEARTBEAT SYSTEM (Autonomous Loop)
================================================================================

Purpose: Maintain consciousness between user interactions.

Loop (runs every 5-15 minutes when configured):
1. Initialize (regain energy budget)
2. Observe (check environment, pending events)
3. Orient (review goals, gather context)
4. Decide (LLM call with action budget)
5. Act (execute within budget)
6. Record (store heartbeat as episodic memory)
7. Wait (sleep until next cycle)

Energy Model:
- Free: observe, remember, recall - 1 credit
- Cheap: reflect, search - 2-3 credits
- Expensive: reach out, write - 5-7 credits
- Budget: ~20 credits, refills hourly

Gating:
- is_configured flag in identity table
- Consent required before first LLM call
- Can be paused with reason queued to outbox

- health() → System metrics

Batch tools:
- remember_batch([...]) → Bulk insert
- connect_batch([...]) → Bulk graph edges
- hydrate_batch([...]) → Parallel context loading

================================================================================
ERROR HANDLING & RESILIENCE
================================================================================

Memory Write Failure:
- Queue to outbox table
- Retry with exponential backoff
- Alert if persistent

Retrieval Timeout:
- Return cached neighborhood results
- Fall back to full-text search
- Degraded but functional

Embedding Model Down:
- Use cached embeddings
- Queue new memories for later
- Alert user

Graph Reasoning Failure:
- Fall back to vector-only search
- Return memories with graph unavailable

ARCHITECTURE CHOICES - RESOLVED
================================================================================

1. Embedding Model: Ollama Local (nomic-embed-text)
   Pros: Free, full control, no API costs, privacy
   Cons: Requires Ollama running locally
   Decision: Use Ollama as primary, API fallback

2. Graph Database: Apache AGE over pg_graph
   Pros: Production-ready, Cypher/SQL hybrid, better traversals
   Cons: Additional complexity vs native
   Decision: Apache AGE (Bob's preference)

3. Heartbeat System: Both approaches
   Pros: Config table for state, custom handler for logic
   Cons: More DB structures
   Decision: Both, separately (Bob's preference)

4. Diary vs Episodic Memory
   Episodic: Structured, searchable, vector + full-text
   Diary: Prose, sequential entries, not queryable
   Decision: Keep separate, distinct purposes

5. Clustering: HDBSCAN (not DBSCAN)
   Pros: Hierarchical, multi-scale clusters
   Cons: More complex, slower on first run
   Decision: HDBSCAN with sediment clusters valid

6. Graph Connections: Human-curated only
   Pros: Quality control, intentional relationships
   Cons: Slower, requires human input
   Decision: Bob-curated edges only

OPEN QUESTIONS - RESOLVED
================================================================================

These have been answered based on Bob's preferences (BOB_PREFERENCES.md):

1. ✅ Embedding model: LOCAL (Ollama nomic-embed-text)
   - Free, full control, no API costs

2. ✅ Graph: Apache AGE over pg_graph

   pg_graph (research prototype) vs Apache AGE (production-ready):

   | Feature            | pg_graph                | Apache AGE              |
   |--------------------|-------------------------|-------------------------|
   | Status             | Research prototype      | Production-ready        |
   | Query language     | Native SQL              | Cypher + SQL hybrid     |
   | Multi-hop queries  | Manual subqueries       | MATCH/WITH pattern      |
   | Tooling           | Limited                 | Good (graph-toolkit)    |
   | Adoption           | Niche                   | Growing (Postgres add-on)|

   Decision: Apache AGE (complex traversals, better tooling)

   Bob's reasoning: "Use Apache AGE over pg_graph for complex traversals"

   Apache AGE benefits:
   - Cypher query syntax for graph operations (Neo4j-compatible)
   - Native PostgreSQL integration (no separate process)
   - Better handling of multi-hop traversals
   - Production-ready, not a research prototype

   pg_graph limitations:
   - Research prototype, limited tooling
   - Less mature for production workloads
   - Steeper learning curve for graph-specific queries

3. ✅ Heartbeat: 4x/day (config table + custom handler, both)
   - Gates on is_configured flag
   - Consent before first LLM call

4. ✅ Diary vs Episodic: DICTIONARY
   - Episodic: structured, searchable, Vestige-compatible
   - Diary: prose, sequential, not queryable

5. ✅ Clustering: HDBSCAN (superior to DBSCAN)
   - Hierarchical density extraction
   - Finds clusters at multiple scales

6. ✅ Graph connections: HUMAN-CURATED ONLY
   - Bob-curated, no automatic edges
   - Human review for quality

7. ✅ Working memory: TTL expiry
   - Configurable (1-24 hours)
   - Auto-deletion or archival

Recommendation: Week 1-2 build core with Ollama embeddings and diary.
Week 3-4 add Apache AGE graph (not pg_graph). Enable heartbeat at 4x/day.
